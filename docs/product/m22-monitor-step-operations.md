# M22 Monitor Step Operations

## Scope

M22 brings M21 conversation run-step observability into the monitor center. Operators can now inspect aggregated step health and filter the unified event stream by event source or step type.

Reused contracts:

```text
GET /api/v1/monitor/overview
GET /api/v1/monitor/events
GET /api/v1/monitor/events/:eventId
```

## Behavior

1. Monitor overview now includes `run_step_summary` and `run_step_breakdown`.
2. Conversation run steps are exposed as unified monitor events with source type `conversation_step`.
3. Step events normalize model, prompt, tool, knowledge, and response phases into `step_type`.
4. Monitor event list supports `source_type` and `step_type` filters.
5. The monitor UI shows a Chinese run-step operations band, step distribution rows, extra filters, and step-aware event detail badges.

## Architecture Notes

M22 does not add a database table or migration. It reads existing `conversation_run.steps` JSON generated by M21 and maps it into monitor response DTOs. Cost in the main monitor summary continues to avoid double-counting step-derived events, while the step summary shows step-level cost separately.
