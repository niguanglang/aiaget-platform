# Product UI Design Image Prompt

Create a high-fidelity product UI design image for the following real frontend route family.

Project context:
- Product/module: 企业 AI Agent 平台，渠道运营信息架构
- Page/route: compatible overview `/channels` plus focused pages `/channels/publish`, provider/account/template/route-rule list and form routes, `/channels/jobs`, `/channels/jobs/:jobId`, `/channels/deliveries`, `/channels/deliveries/:deliveryId`, `/channels/replies`, `/channels/replies/:replyId`, and release governance routes `/channels/release`, `/channels/release/pipeline`, `/channels/release/gate`, `/channels/release/automation`, `/channels/release/self-healing`, `/channels/release/scheduler`, `/channels/release/reports`
- Target users/roles: 渠道管理员、运营人员、Agent 管理员；show read-only states when `channel:publish:view` is present but manage/deploy/disable permissions are missing
- Business goal: 把发布渠道、渠道提供方、账号凭据、消息模板、路由规则、发布任务、投递记录、回复记录拆成职责清晰的真实页面；提供方、账号、模板、路由规则列表只承担查询和行内操作，创建/编辑配置进入独立表单路由；发布任务、投递记录、回复记录列表只承载检索和概览，完整排障信息进入详情页
- Existing frontend stack/design system: Next.js App Router, React Query, Tailwind, shadcn-like `Card`/`Button`/`Input`, `MetricCard`, `StatusBadge`, `EmptyState`, lucide icons
- Existing page shell/layout: console content area with max-width page container, compact enterprise SaaS density, restrained borders/shadows, Chinese UI

Interface contract that must appear in the UI:
- API/service functions: `getPublishChannelOverview`, `enablePublishChannel`, `disablePublishChannel`, `checkPublishChannel`, `listChannelProviders`, `getChannelProvider`, `createChannelProvider`, `updateChannelProvider`, `enableChannelProvider`, `disableChannelProvider`, `deleteChannelProvider`, `listChannelAccounts`, `getChannelAccount`, `createChannelAccount`, `updateChannelAccount`, `enableChannelAccount`, `disableChannelAccount`, `deleteChannelAccount`, `listChannelTemplates`, `getChannelTemplate`, `createChannelTemplate`, `updateChannelTemplate`, `enableChannelTemplate`, `disableChannelTemplate`, `deleteChannelTemplate`, `listChannelRouteRules`, `getChannelRouteRule`, `createChannelRouteRule`, `updateChannelRouteRule`, `enableChannelRouteRule`, `disableChannelRouteRule`, `deleteChannelRouteRule`, `listChannelPublishJobs`, `getChannelPublishJob`, `cancelChannelPublishJob`, `retryChannelPublishJob`, `listChannelDeliveries`, `getChannelDelivery`, `listChannelReplies`, `getChannelReply`
- Main entities and fields: channel name/agent/type/status/health/request count/success rate; account provider/account key/owner/environment/readiness/credential rotation/status; template code/type/language/version/provider/status; route rule priority/match/target/fallback/status; job list fields number/title/type/progress/retry count/updated time; job detail fields timeline/timestamps/error/payload/result; delivery list fields id/provider/account/target/response status/latency/retry count; delivery detail fields request url/headers/body/response/error/trace/conversation/run/external ids; reply list fields id/provider/channel/delivery/external conversation/trace/replied time; reply detail fields external message/internal conversation/run/sender/recipient/content/payload/received/processed time
- Status values/enums: publish channel DRAFT/ACTIVE/DISABLED/ERROR/ARCHIVED and health UNKNOWN/HEALTHY/DEGRADED/UNAVAILABLE; operation statuses ACTIVE/DISABLED/ERROR/DRAFT/EXPIRED/APPROVED/REJECTED; job statuses PENDING/RUNNING/SUCCESS/FAILED/SKIPPED/CANCELED/RETRYING; delivery statuses including SUCCESS/FAILED/RETRYING/PENDING/SKIPPED
- User actions: refresh, keyword/status/provider filters, pagination, enable/disable/delete for configuration pages, channel health check, publish channel enable/disable, row detail expansion, view detail links; job retry/cancel only on `/channels/jobs/:jobId`
- Confirmation flows: configuration enable/disable/delete, publish-channel enable/disable, job cancel/retry, Sender retry/auto-retry/cleanup, release pipeline batch start/full/abort, release gate policy save/evaluation, automation policy save/run, self-healing policy save/run, scheduler run, and release report snapshot creation must open Chinese confirmation dialogs before mutation. Dialogs must show the object name, impact text, cancel button, pending disabled state, and a clear confirm label.
- Required states: loading, empty, API error, action success/error, validation for filters, disabled buttons for permission-denied states, confirmation pending

Design requirements:
- Show `/channels/publish` as a focused publish-channel inventory with summary metrics, status/health chips, endpoint and Agent context, and row actions in a compact “更多” region.
- Show `/channels/providers` as a list-only provider inventory with create/edit route links; show `/channels/providers/create` and `/channels/providers/:providerId/edit` as dedicated provider forms with grouped access endpoint, callback, auth, capabilities, JSON config, and description fields.
- Show `/channels/accounts/create`, `/channels/accounts/:accountId/edit`, `/channels/templates/create`, `/channels/templates/:templateId/edit`, `/channels/route-rules/create`, and `/channels/route-rules/:routeRuleId/edit` as dedicated configuration form pages, while their list routes keep only filters, metrics, compact rows, status/delete actions, and create/edit navigation links.
- Show `/channels/accounts`, `/templates`, `/route-rules` as configuration pages with filters, core list fields, provider context, status actions, and details without crowding the main list.
- Show `/channels/jobs` as an operational queue list with progress/retry/channel summary and a detail link; show `/channels/jobs/:jobId` as the only place for cancel/retry, timeline, timestamps, error, payload and result.
- Show `/channels/deliveries` as a delivery ledger with response/latency/retry/target summary and a detail link; show `/channels/deliveries/:deliveryId` as the only place for request/response/error/Trace/conversation/run/external IDs.
- Show `/channels/replies` as a reply ledger with status/platform/channel/external conversation/Trace coverage summary and a detail link; show `/channels/replies/:replyId` as the only place for message link, sender/recipient, full content and original payload.
- Show `/channels/release` as a lightweight 发布治理总览 with key metrics, release module entry cards and scheduler summary only.
- Show `/channels/release/control` as a focused release-control page with channel picker, approval requirement, approval request/decision workflow, rollout percentage controls and rollback status/actions.
- Show `/channels/release/pipeline`, `/gate`, `/automation`, `/self-healing`, `/scheduler`, and `/reports` as focused release governance pages with shared subnavigation and channel picker where needed; do not show all release governance cards on the overview page.
- Release control page owns publish approval control, approval request, approve/reject decisions, rollout updates and rollback confirmation; pipeline page owns steps, current/recent batches, batch creation form, mark-full confirmation and abort confirmation; gate page owns evaluation, editable gate policy, policy save confirmation and manual evaluation; automation page owns auto-promote status, editable automation policy, policy save confirmation and run action; self-healing page owns rollback recommendation, editable self-healing policy, policy save confirmation and run action; scheduler page owns scheduler run and last-run result; reports page owns report risks and snapshot creation.
- Show high-impact confirmations as compact operational modals, for example 确认启用渠道, 确认停用提供方, 确认删除路由规则, 确认取消任务, 确认申请审批, 确认审批通过, 确认更新灰度, 确认回滚发布, 确认创建发布批次, 确认标记全量, 确认保存推进策略, 确认运行自愈, 确认创建复盘快照.
- Keep `/channels` visually compatible as the full command center, but do not design subroutes as tabs that merely switch the old giant component.
- Use real Chinese product labels, no lorem ipsum, no decorative hero section, no unrelated charts.
